The layer most AI-native developers are missing — and why one-off prompts will never compound.

You've been running Claude Code for months. Or Codex. Or both. You prompt from the terminal, you give good context, and you left ChatGPT-for-coding behind a while ago.

So here's what happened to me last month. I was refining a workflow for weeks — spec-first planning, structured code review, the whole thing. Solid output. Then I looked at what the rest of my team was doing with the exact same tools. Same model. Same terminal. Wildly different results.

Some of the output was, frankly, garbage. Including some of mine.

That's an orchestration problem. And it's the gap in most agentic development workflows right now.

If you just want faster autocomplete, stop here. Cursor and Copilot are great at that. This is for developers who already run CLI agents and can't figure out why their team's output is still all over the place — and what to build above the prompt layer to fix it.

One-off prompts don't compound. Your team uses the same AI tools but gets inconsistent results. The fix is what we call The Orchestration Stack — three layers above prompting: composable skills, governed workflows, and team-wide sharing via your repo.

The plateau you can't see

Here's the difference in a line: a prompt is something you type once. A skill is something you build once and reuse forever. A workflow chains skills into a governed process. That progression — prompt → skill → workflow — is what we call The Orchestration Stack.

Layer 1: Composable skills

Stop writing the same prompt twice.

A skill is a SKILL.md file — structured instructions, metadata, optional scripts. Write it once, the agent loads it automatically. Composition is what makes this work: composable AI skills invoke other skills.

Our spec-first skill produces implementation plans by invoking a three-experts skill — three domain specialists debating the architecture before a single line of code gets written. Our review-changes skill runs a 40-point code review checklist. When it finds observability issues above a threshold, it escalates to an observability-audit skill. Skills calling skills.

I learned this the hard way. I built one massive workflow — planning, implementation, review, deployment, all in one skill. Monolith approach. The context got too wide, instructions too diluted. It was producing goblin work: sloppy, fragile, unmaintainable.

The fix was decomposition. Small skills. Clear boundaries. Same principle we apply to code — single responsibility, composability. Works for AI workflows too.

Keel ships with 40+ skills built this way. spec-first calls three-experts. review-changes dispatches three parallel review agents. creative-content-agent calls humanizer for final polish. You can use them as-is, fork them, or build your own. The composition engine is the point — not any individual skill.

Layer 2: Governed workflows

A workflow chains skills: one skill's output feeds the next. The sequence is enforced — you don't re-prompt between steps.

Feature development in practice:

  1. Prompt refinement — clarifies the request
  2. Spec-first planning — implementation plan with architecture deliberation
  3. Implementation — follows the spec, milestone by milestone
  4. Code review — 40-point automated checklist after each milestone
  5. PR creation — auto-generates the PR with contextual summary

One command. The system handles the sequence. In Keel, these workflows live in a visual canvas — you wire skills together, version them, and publish them for your team. When something breaks, you trace it to the specific skill that failed, not a wall of prompt text.

My unpopular opinion: we're not splitting work into small enough pieces. Most developers run one agent on one task. But you can dispatch subagents — parallel skills handling different concerns simultaneously. Keel's review-changes skill dispatches three parallel subagents (code quality, security/performance, external validation) and merges their findings into a unified report. The cc-cc-powerful-iterations skill cross-validates output between Claude and Codex. The work gets distributed, not serialized.

The speed trap

We spent three months shipping features at what felt like incredible speed. Prompting fast, generating fast, merging fast.

Then came four months of bug hunting. Edge cases the fast prompts missed. Features that weren't spec'd properly. Inconsistencies between developers' implementations.

The governed workflows feel like overhead in week one. But by month two, they're faster — because you're not paying the bug tax later. Small friction now. No four-month cleanup sprint later.

What's still hard

Metrics — biggest open question. We track escaped defects, rework ratio, review cycle time. Keel's /keel-metrics skill automates collection and produces before/after dashboards, but making it rigorous enough for a quarterly review is still harder than it should be.

Memory curation — persistent memory in AI coding sounds great until it goes stale. Someone has to own it, prune it, update it. Like documentation: valuable when maintained, dangerous when neglected.

Granularity — too many tiny skills and you're managing a warehouse. Too coarse and you get the monolith problem — remember the goblin work from earlier. We decompose at roughly 200 lines or when a skill handles more than one concern. Still judgment, not formula.

Where to start

Start with the workflow that hurts the most. If that's inconsistent reviews, run /review-changes on your next PR. If it's specs that drift from implementation, try /spec-first on your next ticket. If your AI keeps forgetting your architecture, run /memory-bank init.

Add a second skill that connects to the first — now you have a workflow. Share it via .claude/skills/ in your repo. Improve through PRs, same as code.

Then let memory compound. Your AI on day 30 knows things it didn't on day 1. By month two, the entire team is running at a quality floor that used to depend on your best engineer being in the room.
